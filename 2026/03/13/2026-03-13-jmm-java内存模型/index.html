# Java内存模型

# 为什么要搞出这么一套模型？

在多线程环境下，由于 CPU 多级缓存、编译器优化以及指令重排序 的存在，
不同线程之间对共享变量的读取和修改可能会出现各种不可预期的问题。

因此 Java 定义了 Java Memory Model（JMM），用来规定多线程之间
变量的读写规则和可见性保证。

JMM 主要围绕三个核心特性展开：

  • 可见性：一个线程修改了变量，其他线程能够及时看到这个修改
  • 原子性：一个操作或代码块执行过程中不可被线程切换打断
  • 有序性：程序执行顺序不会因为指令重排序而破坏

# 抽象出来的内存

JMM 并不是某种真实存在的内存结构，它只是 Java 为了描述多线程之间
变量访问规则而定义的一种 抽象模型。

在这个模型中，JMM 将内存划分为两部分：

  • 主内存（Main Memory）
    存放所有线程共享的变量，例如实例变量和静态变量。

  • 工作内存（Working Memory）
    每个线程私有的一块内存。线程对变量的所有操作（读取、赋值等）
    都必须在自己的工作内存中进行，不能直接操作主内存中的变量。

# happen-before

happen-before这个概念在JSR-133提出，这个概念阐述的是操作之间的内存可见性。如果说
一个操作对另一个操作是可见的，那么就说着两个操作存在happen-before关系。

值得注意的是“一个操作对另一个操作可见”这一句话，并不意味着“一个操作”必须在“另一个操作”的前面发生，只有保证“另一个操作”能看到执行结果就行

  • 程序顺序规则：一个线程中的每个操作，happens- before 于该线程中的任意后续操作。（单线程内执行结果跟按顺序执行是一致的）
  • 监视器锁规则：对一个监视器锁的解锁，happens- before 于随后对这个监视器锁的加锁。（并不仅仅是锁，还包括解锁前的操作，对下一个拿到锁来说是可见的）
  • volatile 变量规则：对一个 volatile 域的写，happens- before 于任意后续对这个 volatile 域的读。
  • 传递性：如果 A happens- before B，且 B happens- before C，那么 A happens- before C。
