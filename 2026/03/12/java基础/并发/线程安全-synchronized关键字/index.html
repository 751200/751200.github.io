# synchronized关键字

用于在多线程环境下保证线程的安全性，它可以保证在同一时刻只有一个线程可以进入被synchronized修饰的方法或代码块。（还好我想了一下，只保证在一个jvm进程内有效）

也就是说特性有两点，这也是线程安全的原理

  • 互斥

    同一时间只有一个线程可以进入代码块

  • 可见性

    必须刷新到主内存，其他线程可见

# 使用

  1. 修饰实例方法，锁对象是实例this

    public synchronized void method() {
}
//等价
public void method() {
    synchronized(this) {
    }
}

  2. 修饰静态方法，锁对象是Class对象

    public static synchronized void method() {
}
//等价
synchronized(ClassName.class) {
}

  3. 修饰代码块，任意对象，关键在于锁的是对象，不是代码

    synchronized(lock) {
    // critical section
}

# 底层实现

获取锁的过程就是获取monitor对象的过程

monitor对象的话里面有三个队列

Monitor
├── Owner (当前持有锁的线程)
├── EntryList (等待锁的线程)
└── WaitSet (调用 wait() 的线程)

当调用notify()方法之后，WaitSet里面的对象会重新进入Entrylist里面，重新竞争锁

JVM字节码中，获取monitor依靠这两个jvm指令
monitorenter
monitorexit

monitorexit指令会存在两个，就像try catch中的finally一样，防止因为错误而不解锁

线程进入
↓
尝试获得Monitor
↓
成功 → 执行代码
失败 → 阻塞
↓
释放Monitor

# 锁升级

无锁
↓（一直是A线程来，不用cas）
偏向锁
↓(出现了B线程,CAS + 自旋获取锁）
轻量级锁
↓(自旋很多次仍然拿不到锁，使用Monitor，会阻塞线程，进入内核态）
重量级锁

锁 说明
偏向锁 只有一个线程
轻量级锁 少量竞争
重量级锁 多线程竞争

# 对象头mark word（optional）

在64位JVM中，大概存：

锁信息
GC年龄
hashcode
线程ID
偏向锁标志

加锁就是修改对象头的线程ID
